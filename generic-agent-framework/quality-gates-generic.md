# Quality Gates (Generic)

Coding agents must keep the pipeline green. These gates are mandatory once the implementation workspace exists.

## Standard Completion Commands

Configure and run your project's equivalent quality gates:

```sh
# Linting
[YOUR_LINT_COMMAND]  # e.g., npm run lint, pnpm lint, pylint

# Type checking
[YOUR_TYPECHECK_COMMAND]  # e.g., tsc --noEmit, npm run type-check

# Unit tests
[YOUR_TEST_COMMAND]  # e.g., npm test, pytest, go test

# Test coverage
[YOUR_COVERAGE_COMMAND]  # e.g., npm run test:coverage, pytest --cov

# Build
[YOUR_BUILD_COMMAND]  # e.g., npm run build, cargo build

# Contract/schema validation
[YOUR_CONTRACT_COMMAND]  # e.g., npm run contract:check, openapi validate

# Database migrations (if applicable)
[YOUR_MIGRATE_COMMAND]  # e.g., npm run db:migrate:check
```

Generated artifacts required by these commands must be generated by the command graph itself, not by a developer's local machine state. For example, generated code must be produced by the build system, not committed as pre-generated files.

## Required Extra Gates by Change Type

| Change type | Extra gates |
| --- | --- |
| API contract | API contract/OpenAPI generation, generated client compilation, API integration tests |
| Auth/permission/visibility | Matrix tests for all roles, scopes, and permission combinations |
| Database migration | Clean DB migrate, previous-state migrate, seed load, rollback/forward tests |
| Frontend screen/page | Navigation smoke test, loading/error/empty states, accessibility checks |
| Backend service | Integration tests, permission enforcement tests, state management tests |
| UI/theming | Design token usage check, no hardcoded styling, contrast/accessibility checks |
| App shell/runtime | App launch smoke checks, demo/test mode handling, production mode enforcement |
| Data privacy | Archive/export/erasure tests, audit trail checks, anonymization tests |
| Notifications | Preference filtering, audience safety tests (don't leak data to wrong users) |
| Dependency upgrade | Lockfile update, changelog check, full lint/typecheck/test/build |
| Generated dependency | Generator target in build system, clean install checks |

## Coverage Gate

- **Minimum coverage**: 80% for statements, branches, functions, and lines in every implemented app/library
- **Enforcement**: Coverage must be enforced in CI
- **Quality**: New or changed business logic needs meaningful tests, not just incidental coverage
- **Exceptions**: Generated files and framework bootstrap files may be excluded only through narrow, documented patterns
- **Security code**: Security-sensitive shared libraries should exceed the minimum where practical

## Red Pipeline Rule

- Do not finish a task with a failing required gate
- Do not skip, weaken, or delete tests to make the pipeline green
- If a failure is unrelated to your change, document evidence and still avoid merging until human owner accepts it
- If a command is unavailable because repo is not bootstrapped yet, add the missing script/target during foundation phase

## Quality Bar

- **Security/Access control**: Tested, not assumed. Permission and visibility behavior is verified by tests
- **Test quality**: New behavior has the narrowest useful test that would fail without the change
- **Generated files**: Updated by documented commands, not hand-edited
- **Dependencies**: Lockfiles and generated artifacts stay aligned with package manifests and schema changes
- **No duplication**: Reuse existing code before creating new shared abstractions
- **Final response**: Must list quality gates run and the result

## Coverage Targets by Component Type

| Component | Min Coverage | Type | Notes |
|-----------|--------------|------|-------|
| Auth/permissions | 85% | Unit + Integration | Security-critical code |
| API endpoints | 80% | Unit + Integration | Test success, errors, permissions |
| Data access | 80% | Unit + Integration | Test queries, filtering, access control |
| Business logic | 80% | Unit | Test algorithms, calculations, workflows |
| UI components | 70% | Unit + Visual | Test rendering, props, callbacks |
| Utilities | 80% | Unit | Test edge cases, error handling |
| Infrastructure | 60% | Integration | Test configuration, setup, teardown |

## Recommended Testing Approach

### Unit Tests
- Test single function/method in isolation
- Use mocks for external dependencies
- Fast execution (<1 second per test)
- Coverage: 80%+ of business logic

### Integration Tests
- Test multiple components working together
- Use real or in-memory databases
- Test API endpoints with requests/responses
- Test permission enforcement
- Coverage: Critical workflows, permission boundaries

### E2E Tests
- Test complete user workflows
- Use staging environment or local server
- Smoke tests for critical paths
- Keep to essential workflows (slower)

### Smoke Tests
- Quick verification that system starts
- Basic functionality works
- Run after deployment

## Continuous Integration Best Practices

✅ **Do:**
- Run all gates on every commit to main
- Run fast gates (lint, typecheck) on PRs
- Run full test suite on merge to main
- Track coverage trends
- Block merges if gates fail
- Fail fast (stop on first failure)

❌ **Don't:**
- Skip required gates
- Weaken tests to make gates pass
- Commit generated files without regenerating
- Ignore flaky tests
- Delete tests to reduce coverage debt
